1. Executive Summary

Peoples Bakers is a full-stack web application for a Sri Lankan bakery business. Customers browse products, build a cart, place orders (WhatsApp, cash-on-delivery, or PayHere online payment), manage accounts, submit feedback, and locate outlets on a map. Staff use a protected admin dashboard for order fulfilment, catalog management, and analytics.

Key characteristics

AspectDetail
ArchitectureMonolithic Express server serving static frontend + JSON REST API
FrontendMulti-page application (MPA) — HTML/CSS/vanilla JS, no React/Vue build step
BackendNode.js 22.5+ with Express 4, modular route handlers
PersistenceSQLite file via Node built-in node:sqlite — no MySQL/Postgres required
AuthJWT in httpOnly cookie (pb_token), bcrypt password hashing, optional Google OAuth
PaymentsPayHere (Sri Lanka) with server-side hash verification and webhook notify URL
LanguagesEnglish, Sinhala (සිං), Tamil (த) — client-side i18n with JSON dictionaries
MobileResponsive CSS (30+ breakpoints), hamburger nav, progressive image loading
Critical constraint: This is not a static site. Opening HTML files directly (file://), VS Code Live Server, GitHub Pages, or PHP-only hosting will break login, orders, admin, and payments. A Node.js process must run continuously in production.

5. Local Development Setup

Prerequisites

  • Node.js 22.5 or newer — nodejs.org
  • Git or ZIP of project folder
  • Terminal (PowerShell on Windows)

Commands

cd "C:\path\to\Peoples-Bakers-master"
npm install          # First time only — installs node_modules
npm start            # Starts Express on http://localhost:3000
npm run dev          # Auto-restart on file changes (Node --watch)
npm run verify       # Health checks (server must be running)
npm run seed         # Re-seed products/admin (no duplicates)
npm run promote-admin -- user@email.com
Success indicator: Terminal prints Peoples Bakers server running at http://localhost:3000. Keep the terminal open — closing it stops the server. Use Ctrl+C to stop intentionally.

First-run seed data

  • Admin account created automatically (credentials printed in terminal)
  • Default dev: admin@peoplesbakers.lk / admin123 — change before production
  • Products imported from seed data into SQLite

8. Authentication & Authorization

8.1 JWT cookie model

  • Cookie name: pb_token
  • Flags: httpOnly, sameSite: lax, secure: true in production
  • Expiry: 7 days
  • Payload: user id, name, email, phone, role (customer | admin)

8.2 Role-based access

RoleAccess
customerOwn profile, own orders, public pages
adminAll admin API routes, order management, products, feedback, users, settings

8.3 Security hardening (implemented)

  • Production JWT_SECRET: Minimum 32 characters required; server exits if missing
  • Stale JWT role fix: Middleware re-loads user from DB on each request — promoted/demoted roles apply immediately
  • Admin seed password: ADMIN_SEED_PASSWORD required in production (no default admin123)
  • Secret admin URL: ADMIN_PATH env (default pb-office.html) — not linked from public nav
  • Server-side pricing: Client-submitted prices ignored; catalog is source of truth
  • Phone discount: Only applied when logged-in user's phone matches checkout phone

9. Order & Payment Lifecycle

9.1 Order creation

  1. Frontend sends cart items + customer details to POST /api/orders
  2. Server maps product IDs to catalog entries; rejects unknown or zero-qty items
  3. Discount calculated server-side (discounts.js)
  4. Order persisted with timestamps, payment method, delivery mode
  5. Optional confirmation email via mailer.js

9.2 PayHere integration

  • Checkout endpoint returns merchant_id, hash, amount, return/cancel/notify URLs
  • User completes payment on PayHere hosted page
  • PayHere POSTs to /api/payments/payhere/notify — MD5 hash verified server-side
  • Failed payment no longer auto-cancels order (fixed) — staff can follow up manually
  • Production defaults to live mode unless PAYHERE_SANDBOX=true

10. Database Layer (SQLite)

10.1 Design pattern

Collections: users, orders, feedback, products, settings. Each table: (id TEXT PK, createdAt TEXT, doc TEXT) where doc is JSON. This document-store pattern keeps route code simple while using a real SQL database.

10.2 Operations

  • WAL journal mode for concurrent read performance
  • Write transactions for crash-safe multi-step updates
  • Legacy JSON files auto-migrated on first run
  • Override path: DB_PATH=/persistent/peoples-bakers.db

10.3 Reset

Stop server → delete server/data/ folder → npm start (re-seeds)

12. Internationalization (i18n)

  • Engine: js/i18n.js loads js/translations/{en,si,ta}.json
  • Markup: data-i18n="section.key" on HTML elements

13. PWA, Caching & Performance

  • service-worker.js: Caches static assets; network-first strategy for main JS/CSS (prevents stale code)
  • Cache versioning: Bumped on releases (e.g. peoples-bakers-v147)
  • Asset query strings: CSS/JS ?v=N for cache busting
  • Config API: Contact numbers loaded dynamically — single source of truth

14. Mobile Responsive Design

Majority of Sri Lankan e-commerce traffic is mobile. The site implements responsive layout, touch-friendly controls, and network-aware image loading.

14.1 CSS breakpoints

css/style.css includes 30+ media queries at 768px, 900px, 1024px, etc. Grids collapse to single column on small screens.

14.2 Hamburger navigation (≤900px)

  • js/main.js — setupMobileNav()
  • Side drawer with backdrop; closes on link click, backdrop tap, Escape key
  • body.nav-open { overflow: hidden } prevents background scroll

14.3 Progressive image loading (3G / slow networks)
